# Architecture Decision Records

## Purpose

Architecture Decision Records preserve the reasoning behind material technical and documentation decisions. They make trade-offs discoverable and prevent future contributors from repeating settled debates without new evidence.

## ADR Numbering Strategy

ADRs use sequential four-digit identifiers:

```text
ADR-0001-title.md
ADR-0002-title.md
ADR-0003-title.md
```

Numbers are never reused. Superseded ADRs remain in the repository and link to the replacing decision.

## ADR Lifecycle

- `Proposed`: The decision is under discussion.
- `Accepted`: The decision governs future work.
- `Superseded`: A later ADR replaces the decision.
- `Rejected`: The decision was considered but not adopted.

## Current ADR Index

No implementation ADRs exist yet. The first ADR should define documentation governance and the threshold for introducing implementation code.

## Required ADR Content

Each ADR should include:

- Status.
- Context.
- Decision.
- Alternatives considered.
- Consequences.
- Review date or review trigger.
