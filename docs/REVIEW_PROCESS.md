# Review Process

## Purpose

This document defines the professional review process for AIFlow OS Engineering Handbook changes. It applies to governance documents, roadmap updates, standards, ADRs, diagrams, volume structures, chapters, templates, prompts, and GitHub community files.

See [../diagrams/documentation-review-workflow.mmd](../diagrams/documentation-review-workflow.mmd).

## Architecture Review

Architecture review checks whether the change aligns with the constitution, roadmap, architecture overview, ADRs, and future platform boundaries. It should identify missing decisions, unclear boundaries, dependency issues, and unsupported implementation assumptions.

## Technical Review

Technical review checks accuracy, completeness, system reasoning, terminology, requirements, diagrams, and operational implications. It should confirm that claims are specific enough to guide future work.

## Editorial Review

Editorial review checks clarity, organization, heading structure, grammar, tone, concision, and handbook readability. The goal is professional engineering documentation, not marketing copy.

## Consistency Review

Consistency review checks glossary usage, file naming, cross-references, roadmap alignment, changelog updates, master context updates, and diagram synchronization.

## Quality Review

Quality review confirms that the change has no placeholders, no filler, no unsupported claims, no broken structure, and no missing required sections for its document type.

## Final Approval

Final approval confirms that all required review categories are complete, material decisions are captured in ADRs, and the change is ready to merge or publish.

## Publishing Approval

Publishing approval confirms version metadata, changelog, master context, release scope, diagrams, and public-facing GitHub templates are consistent. Publishing approval is required for versioned releases.
